# 실습: 테스트 환경 설정 및 실행 오류 해결

단위 테스트 실습 중 빈번하게 발생하는 라이브러리 미설치, 경로 인식 문제, 프레임워크 충돌 등을 해결하기 위한 가이드입니다.

---

## 1. 라이브러리 및 의존성 관련 오류

### 현상: 'ModuleNotFoundError' 또는 'Package not found'
* **원인:** pytest, mockito 등 테스트 프레임워크가 현재 가상환경에 설치되지 않았을 때 발생합니다.
* **해결 방법 (Python):**
    - 터미널에서 `pip list`를 입력하여 `pytest`가 있는지 확인합니다.
    - 설치되지 않았다면 `pip install pytest pytest-mock`을 실행합니다.
    - Poetry 사용 시: `poetry add --group dev pytest`
* **해결 방법 (Java):**
    - `pom.xml`(Maven) 또는 `build.gradle`(Gradle)에 JUnit5/Mockito 의존성이 포함되었는지 확인합니다.
    - IDE(IntelliJ 등)에서 'Reload Project'를 수행하여 라이브러리를 다시 내려받습니다.

---

## 2. 경로 및 임포트 인식 오류

### 현상: 'No module named src' 또는 테스트 파일 탐색 실패
* **원인:** 테스트 실행 위치와 소스 코드의 루트 경로가 일치하지 않거나, 파이썬이 패키지를 인식하지 못하는 경우입니다.
* **해결 방법 (Python):**
    - **실행 위치 확인:** 항상 프로젝트의 최상위 루트 폴더(Root Directory)에서 테스트를 실행해야 합니다.
    - **PYTHONPATH 설정:** 터미널에서 `export PYTHONPATH=$PYTHONPATH:.` (Linux/macOS) 또는 `$env:PYTHONPATH = "."` (PowerShell) 명령어를 사용해 현재 경로를 인식시킵니다.
    - **__init__.py 확인:** `src/` 폴더와 `tests/` 폴더 내에 빈 `__init__.py` 파일이 있는지 확인하여 패키지로 인식되도록 합니다.
* **해결 방법 (Java):**
    - `src/main/java`와 `src/test/java` 폴더 구조가 표준에 맞게 구성되었는지 확인합니다.

## 3. GitHub Copilot 관련 이슈

### 현상: /tests 명령어가 동작하지 않거나 엉뚱한 프레임워크 코드를 생성함
* **원인:** 에디터에 열린 파일이 부족하여 Copilot이 프로젝트의 기술 스택을 오판한 경우입니다.
* **해결 방법:**
    - **컨텍스트 제공:** `main.py`나 `pom.xml` 등 설정 파일을 에디터 상단 탭에 열어둔 상태에서 다시 시도합니다.
    - **명시적 지시:** "이 프로젝트는 pytest를 사용해. pytest 기반으로 테스트 코드를 작성해줘"라고 채팅창에 명시합니다.
    - **파일 선택:** 테스트할 함수를 정확히 드래그(Highlight)한 후 `/tests`를 입력했는지 확인합니다.

## 4. 실행 권한 및 환경 이슈

### 현상: 'pytest' is not recognized as an internal or external command
* **원인:** 설치는 되었으나 실행 경로(PATH)에 등록되지 않은 경우입니다.
* **해결 방법:**
    - `python -m pytest` 명령어를 사용하여 파이썬 모듈로서 직접 실행합니다.
    - 가상환경(venv)이 활성화(Activate)되어 있는지 터미널 왼쪽의 프롬프트를 확인합니다.

---

## 5. Mocking 관련 실행 오류

### 현상: AttributeError: 'Mock' object has no attribute 'xxx'
* **원인:** Mock 객체를 생성할 때 실제 클래스의 인터페이스와 일치하지 않는 메서드명을 호출했거나, Stubbing 설정이 잘못된 경우입니다.
* **해결 방법:**
    - `return_value`와 `side_effect`의 철자를 확인합니다.
    - `spec` 옵션을 사용하여 실제 클래스 구조를 기반으로 Mock을 생성하도록 프롬프트를 수정합니다. (예: `MagicMock(spec=PaymentGateway)`)
