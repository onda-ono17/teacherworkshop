# 실습: Mocking과 Fixture 생성

단위 테스트를 복잡하게 만드는 가장 큰 원인은 '의존성(Dependencies)'입니다. 데이터베이스 연결, 서드파티 API 통신, 외부 파일 시스템 등의 인프라 로직은 순수한 비즈니스 로직 테스트를 방해합니다.

---

## 1. 왜 Mocking이 필요한가?
*   **속도 향상:** DB나 네트워크를 거치지 않고 테스트가 수 밀리초(ms) 단위로 완료됩니다.
*   **독립성:** 외부 서버가 다운되더라도 내 코드의 테스트는 통과해야 합니다.
*   **테스트 일관성:** 실제 운영 DB를 훼손하지 않으면서 멱등성(Idempotent)을 유지합니다.

## 2. Copilot을 활용한 Mock 데이터 및 객체 자동화

Copilot은 복잡한 Mock/Stub 클래스나 가짜 데이터 셋인 픽스처(Fixture)를 순식간에 만들어내는 데 매우 탁월합니다.

### A. 대량의 Fixture (더미 데이터) 생성
개발자가 테스트를 위해 손으로 여러 줄의 딕셔너리나 JSON 객체를 타이핑할 필요가 없습니다.
> **프롬프트 예시:** `@workspace #file:user_model.py 이 스키마를 바탕으로,pytest 모듈에서 활용할 수 있는 가짜 유저 데이터(Fixture) 리스트 10개를 한국어 이름으로 만들어줘.`

### B. 의존성 격리 (Mock Object)
서비스 객체가 특정 인터페이스(예: S3 업로더, 외부 결제 서비스)를 의존할 때 가짜 객체를 만듭니다. Python의 경우 `unittest.mock.patch`, Java의 경우 `Mockito`, JavaScript의 경우 `Jest.mock` 등의 사용법을 정확히 숙지하고 있습니다.
> **프롬프트 예시:** `OrderService를 테스트하려는데, 결제를 담당하는 TossPayment 클래스의 API 호출을 Mocking하는 모의 객체를 만들어줘. Status는 'SUCCESS'로 고정해서 반환하게 해.`

---

## 3. 유의 사항
*   테스트 코드 생성 시 Copilot이 어떤 라이브러리 프레임워크를 기준으로 Mocking을 했는지 확인해야 합니다 (예: Pytest의 `monkeypatch` 를 쓸 것인지 기본 모듈인 `unittest.mock` 을 쓸 것인지 패러다임을 한정해주는 것이 좋습니다).
