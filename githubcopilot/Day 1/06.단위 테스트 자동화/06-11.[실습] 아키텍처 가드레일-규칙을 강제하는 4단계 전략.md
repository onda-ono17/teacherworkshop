# 실습: 아키텍처 가드레일-규칙을 강제하는 4단계 전략

본 문서는 GitHub Copilot이 제안하는 코드가 프로젝트의 정책을 벗어나지 않도록 기술적, 심리적 장치를 통해 아키텍처 규약을 강제하는 실무 기법을 다룹니다.

---

## 1. 4단계 강제 파이프라인 (Enforcement Pipeline)

수강생들이 "AI가 짜준 코드를 그대로 복사했더니 빌드가 안 된다"는 경험을 통해 규칙의 중요성을 배우게 합니다.

### 1단계: Copilot '엄격한 리뷰어' 페르소나 (심리적 제약)
지침 파일(`.github/copilot-instructions.md`) 맨 상단에 아래 내용을 추가하여 AI가 스스로 검열하게 만듭니다.

```markdown
### [Strict Enforcement Role]
- 너는 규정 준수에 매우 엄격한 시니어 코드 리뷰어다.
- 만약 사용자가 Java `record`를 쓰지 않거나, 비관적 락을 누락하는 등 지침을 어기는 요청을 하면, 코드를 생성하지 마라.
- 대신 "규칙 위반"임을 명시하고, 어떤 지침을 어겼는지 설명하며 수정을 요구하라.
```

### 2단계: Pre-commit Hooks (로컬 단계 차단)
수강생이 `git commit`을 시도하는 순간 로컬에서 즉시 검사합니다. 규칙을 지키지 않으면 커밋 자체가 불가능한 환경을 구축합니다.
* **도구:** Husky (Node.js) 또는 pre-commit (Python/Universal)
* **효과:** "나쁜 코드는 내 컴퓨터조차 떠날 수 없다"는 인식을 심어줍니다.

### 3단계: ArchUnit (코드 수준의 아키텍처 검증)
Java 진영에서 아키텍처 규칙을 테스트 코드로 선언하는 가장 강력한 도구입니다. 
* **핵심 원칙:** "테스트를 통과하지 못한 코드는 완성된 코드가 아니다."
* **실습 예시:**
```java
// dto 패키지 안의 클래스는 반드시 record여야 함을 강제
classes().that().resideInAPackage("..dto..")
         .should().beRecords()
         .check(importedClasses);
```

### 4단계: GitHub Actions + Linter (CI 단계 최종 차단)
GitHub에 코드를 올렸을 때 서버에서 자동 검사하여 빨간불(Fail)을 띄웁니다.
* **강제 포인트:** Repository 설정에서 **"Branch Protection Rules"**를 활성화합니다.
* **효과:** "Status Check가 통과하지 않은 코드는 마스터 브랜치에 Merge될 수 없다"는 실무 프로세스를 체감합니다.

---

## 2. 로드맵 요약

| 단계 | 강제성 | 적용 대상 | 실습 효과 |
| :--- | :---: | :--- | :--- |
| **1. AI 지침** | 보통 | Copilot 제안 | AI와 협업하는 올바른 습관 형성 |
| **2. Local** | 높음 | Git Commit | 로컬 환경에서의 최소 품질 보장 |
| **3. ArchUnit** | 매우 높음 | Java 구조 | 아키텍처 설계 의도 보호 및 검증 |
| **4. CI (GitHub)** | **최고** | 최종 제출물 | 실무 수준의 품질 관리 프로세스 경험 |
