# 실습: Mock 객체 활용 및 코드 스니펫

단위 테스트의 가장 큰 장애물인 외부 의존성(DB, 외부 API)을 해결하기 위해 Mocking 기술을 학습합니다. 실제 환경을 구축하지 않고도 격리된 환경에서 비즈니스 로직만을 정교하게 검증하는 방법을 익힙니다.

---

## 1. Mock 객체는 왜 필요한가?

단위 테스트는 격리가 핵심입니다. 하지만 현실의 코드는 데이터베이스나 외부 서비스와 밀접하게 연결되어 있습니다. 이를 위해 가짜 객체(Mock)를 사용합니다.

* 외부 시스템과의 절연: 실제 데이터베이스에 연결하거나 네트워크 통신을 하지 않고도 로직을 검증할 수 있습니다. 테스트 실행 속도가 비약적으로 빨라집니다.
* 에러 상황 시뮬레이션: 외부 결제 API가 500 에러를 반환하거나, 네트워크 타임아웃이 발생하는 상황 등 실제 환경에서 재현하기 어려운 에러 케이스를 강제로 만들어 테스트할 수 있습니다.
* 비용 및 보안: 실제 결제 API를 호출하여 비용이 발생하는 것을 방지하고, 테스트 과정에서 실제 고객 데이터가 유출되는 위험을 차단합니다.

---

## 2. Python: unittest.mock 활용

파이썬 표준 라이브러리인 unittest.mock과 MagicMock을 사용하여 외부 의존성을 대체합니다.

**파일명: test_order_service.py**

from unittest.mock import MagicMock
import pytest

def test_order_process_with_mock(order_service, payment_gateway):
    # 1. Mock 객체 설정
    # 결제 게이트웨이의 process 메서드가 호출되면 항상 True를 반환하도록 설정합니다.
    payment_gateway.process = MagicMock(return_value=True)
    
    # 2. 로직 실행
    # 실제 결제 API를 호출하지 않고 Mock의 설정된 반환값을 사용합니다.
    result = order_service.complete_order(order_id=123)
    
    # 3. 검증
    # 최종 결과가 성공인지 확인하고, 결제 메서드가 정확히 호출되었는지 체크합니다.
    assert result == "SUCCESS"
    payment_gateway.process.assert_called_once()

## 3. Java: Mockito 활용

자바 진영의 표준적인 Mock 프레임워크인 Mockito를 활용하여 의존성을 주입하고 동작을 정의합니다.

**파일명: OrderServiceTest.java**

import static org.mockito.Mockito.*;
import static org.junit.jupiter.api.Assertions.*;

import org.junit.jupiter.api.Test;
import org.junit.jupiter.api.extension.ExtendWith;
import org.mockito.InjectMocks;
import org.mockito.Mock;
import org.mockito.junit.jupiter.MockitoExtension;

@ExtendWith(MockitoExtension.class)
class OrderServiceTest {

    @Mock
    private PaymentGateway paymentGateway; // 가짜 결제 게이트웨이

    @InjectMocks
    private OrderService orderService; // Mock이 주입될 실제 서비스

    @Test
    void testCompleteOrder() {
        // 1. Mock 설정 (Stubbing)
        // 어떤 금액이 들어오든 결제 process는 true를 반환하도록 시뮬레이션합니다.
        when(paymentGateway.process(anyDouble())).thenReturn(true);

        // 2. 실행
        String result = orderService.completeOrder(123L);

        // 3. 검증
        // 반환값이 기대와 일치하는지 확인하고, Mock 객체가 의도대로 호출되었는지 검증합니다.
        assertEquals("SUCCESS", result);
        verify(paymentGateway, times(1)).process(anyDouble());
    }
}

## 4. 실습 핵심 포인트

* Behavior Verification: 단순히 결과값(Return Value)만 확인하는 것이 아니라, 의존하고 있는 객체의 메서드가 특정 횟수만큼 올바른 인자로 호출되었는지 확인하는 것이 중요합니다.
* Over-Mocking 주의: 너무 많은 객체를 Mock으로 만들면 테스트가 실제 동작과 동떨어질 수 있습니다. 오직 제어할 수 없는 외부 의존성(DB, API 등)에 대해서만 Mock을 적용하는 것이 권장됩니다.

    - 프롬프트: "PaymentGateway가 실패 응답을 보낼 때 OrderService가 어떻게 예외를 처리하는지 테스트하는 Mock 코드를 작성해줘."
