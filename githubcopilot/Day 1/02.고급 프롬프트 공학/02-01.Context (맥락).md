# 프롬프트의 3대 기둥: Context (맥락)

맥락(Context)은 "무엇을 만들고 있는가?"에 대한 해답을 제시하는 단계입니다. GitHub Copilot은 단순히 현재 파일의 텍스트를 읽는 것을 넘어, 개발자의 의도를 파악하기 위해 주변 환경을 적극적으로 참고합니다.

## 1. 주변 탭 참조 알고리즘 (Neighboring Tabs)
GitHub Copilot의 핵심 기술 중 하나는 현재 편집 중인 파일뿐만 아니라, IDE 상단에 **열려 있는 다른 탭(Neighboring Tabs)**의 내용을 참고한다는 점입니다. 

* **작동 원리:** Copilot은 사용자가 프롬프트를 입력할 때, 현재 열려 있는 탭들 중에서 문맥적으로 관련성이 높은 파일을 분석하여 프롬프트의 일부로 포함시킵니다.
* **중요성:** 아무리 현재 파일에서 상세하게 설명하더라도, 프로젝트의 전체적인 아키텍처나 공통 유틸리티의 구조를 Copilot이 모르면 프로젝트의 일관성을 해치는 코드를 제안할 수 있습니다.

## 2. 맥락 확장을 위한 전략적 환경 구성
생산성을 극대화하기 위해서는 Copilot이 참고할 수 있는 '재료'를 미리 IDE 환경에 배치해야 합니다.

* **관련 파일 오픈:** 데이터 구조 정의서(Models), 공용 유틸리티(Utils), 혹은 API 명세서 파일을 탭에 열어두는 것만으로도 Copilot은 해당 파일의 함수명이나 변수명을 정확히 인용하여 제안합니다.
* **관련성 기반 필터링:** Copilot은 모든 열린 탭을 무작위로 읽지 않습니다. 현재 작업 중인 코드와 의미적으로 유사한 파일(예: 같은 도메인 로직을 다루는 파일)을 우선순위에 둡니다.

## 3. 맥락 유무에 따른 제안 품질의 차이

**[맥락이 부족한 경우]**
* 상황: `User` 클래스 정의가 담긴 `models.py`를 탭에서 닫아둔 상태
* 결과: Copilot은 일반적인 `User` 클래스 구조(예: id, name, email)를 가정하여 코드를 작성합니다. 실제 프로젝트에서 사용하는 `user_uuid`나 `created_at` 같은 필드를 놓칠 확률이 높습니다.

**[맥락이 풍부한 경우]**
* 상황: `models.py`와 데이터베이스 연결 설정을 담은 `db_config.py`를 탭에 열어둔 상태
* 결과: Copilot은 프로젝트의 실제 DB 필드명과 설정된 라이브러리(예: SQLAlchemy, Prisma)의 문법을 정확히 반영한 최적의 쿼리 로직을 제안합니다.

## 4. 결론: 개발자의 역할
개발자는 단순히 타이핑을 하는 사람이 아니라, **Copilot이 올바른 판단을 내릴 수 있도록 최적의 맥락(Context)을 제공하는 감독관** 역할을 수행해야 합니다. 관련 있는 파일을 열어두는 사소한 습관이 프롬프트 엔지니어링의 시작입니다.
