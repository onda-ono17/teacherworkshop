# 프롬프트의 3대 기둥: Paradigm (패러다임/스타일)

패러다임(Paradigm)은 "어떤 방식으로 코드를 구현할 것인가?"에 대한 설계 지침입니다. 개발자가 선호하는 특정 프레임워크, 라이브러리, 혹은 코딩 스타일을 Copilot에게 명시하여 프로젝트의 일관성을 유지하는 단계입니다.

## 1. 프레임워크 및 라이브러리 지정
동일한 기능을 구현하더라도 사용하는 도구에 따라 코드의 구조는 완전히 달라집니다. Copilot에게 사용할 도구를 명확히 지시함으로써 불필요한 수정을 줄일 수 있습니다.

* **예시 (FastAPI vs Flask):**
    * **FastAPI 지시:** `# FastAPI를 사용하여 비동기(async) 방식으로 사용자 조회 엔드포인트 작성`
    * **Flask 지시:** `# Flask를 사용하여 동기 방식으로 사용자 조회 라우트 작성`
* **효과:** 비동기 처리(`await/async`)의 포함 여부와 프레임워크 고유의 데코레이터 문법을 정확하게 반영합니다.

## 2. 코딩 스타일 및 설계 패턴 설정
프로젝트에서 지향하는 프로그래밍 패러다임을 주석이나 초기 코드로 설정할 수 있습니다.

* **함수형 프로그래밍(Functional Programming):**
    * 지시 예시: `# 고차 함수(map, filter, reduce)를 사용하여 리스트 데이터 가공`
    * 결과: 반복문(for/while) 대신 선언적인 함수형 코드를 제안합니다.
* **객체 지향 프로그래밍(Object-Oriented Programming):**
    * 지시 예시: `# Repository 패턴을 적용하여 데이터베이스 접근 로직을 클래스로 캡슐화`
    * 결과: 인터페이스 정의, 클래스 상속 및 의존성 주입 구조를 갖춘 코드를 제안합니다.

## 3. 패턴 매칭을 통한 스타일 유도 (Leading by Example)
Copilot은 이전 코드의 패턴을 학습하여 뒤에 올 코드를 예측합니다. 패러다임을 직접 명시하지 않더라도, 첫 번째 함수를 특정 스타일로 작성하면 이후 코드도 그 스타일을 따릅니다.

* **작업 방법:**
    1.  파일의 첫 번째 함수를 원하는 스타일(예: 화살표 함수, 타입 정의 포함 등)로 직접 작성합니다.
    2.  두 번째 함수부터는 Copilot이 앞선 코드의 문법과 구조를 복제하여 제안합니다.
* **핵심:** 코드의 '톤앤매너'를 일치시키는 가장 강력한 방법입니다.

## 4. 패러다임 지정 유무에 따른 차이 비교

| 항목 | 패러다임 미지정 시 | 패러다임 명시 시 |
| :--- | :--- | :--- |
| **코드 구조** | 일반적이고 범용적인 구조 | 프로젝트 아키텍처에 최적화된 구조 |
| **라이브러리 활용** | 대중적인 라이브러리 임의 선택 | 지정된 특정 라이브러리 문법 준수 |
| **유지보수성** | 일관성 없는 코드로 인해 리팩토링 필요 | 기존 코드 베이스와의 높은 일관성 유지 |

## 5. 결론: 스타일의 명확화
Paradigm을 설정하는 것은 Copilot에게 **'우리 팀의 코딩 규칙'**을 전달하는 것과 같습니다. 구현하려는 기능의 결과물뿐만 아니라, 그 결과를 도달하기 위한 방법론을 프롬프트에 포함하십시오.
