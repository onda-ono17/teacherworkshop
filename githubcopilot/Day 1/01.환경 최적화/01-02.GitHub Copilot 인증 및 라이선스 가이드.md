# GitHub Copilot 인증 및 라이선스 가이드

GitHub Copilot은 사용자의 라이선스 유형에 따라 인증 경로가 달라집니다. 특히 기업 환경에서는 조직의 보안 정책(SSO)에 따라 추가적인 인증 단계가 포함될 수 있습니다.

## 1. 계정 유형별 인증 차이점

| 구분 | 개인용 (Individual) | 기업용 (Business/Enterprise) |
| :--- | :--- | :--- |
| **로그인 주체** | 개인 GitHub 계정 | 조직(Organization)에 소속된 계정 |
| **인증 방식** | 표준 GitHub OAuth | **SSO(SAML) 인증** 필수 (회사 포털 로그인) |
| **라이선스 관리** | 본인이 직접 구독 및 결제 | 조직 관리자가 라이선스 할당 및 회수 |
| **정책 제어** | 기본 설정 사용 가능 | 공개 코드 매칭 차단, 원격 분석 데이터 공유 등 강제 설정 가능 |
| **인증 유효 기간** | 세션 만료 시까지 장기간 유지 | 보안 정책에 따라 주기적인 재인증 요구 가능 |

- GitHub Copilot 가격 비교

## 2. 인증 워크플로우 (Authentication Flow)

Copilot 확장 프로그램은 다음과 같은 단계를 통해 라이선스를 검증합니다.

1. **IDE 로그인:** VS Code 등에서 "Sign in to GitHub" 클릭
2. **브라우저 리다이렉트:** GitHub.com 로그인 페이지로 이동
3. **SSO 확인(기업용):** 기업용 계정인 경우, 회사의 ID 공급자(Okta, Azure AD 등) 페이지로 이동하여 추가 인증 진행
4. **토큰 발급:** 인증 성공 시 IDE로 보안 토큰(OAuth Token) 전달
5. **라이선스 체크:** GitHub 서버가 해당 계정에 활성화된 Copilot 구독이 있는지 최종 확인

## 3. 인증 문제 발생 시 주요 조치 방법 (Troubleshooting)

로그인이 반복적으로 풀리거나, 라이선스가 없다는 오류(No Copilot subscription found)가 발생할 경우 아래 단계를 순서대로 진행합니다.

### 1단계: 계정 로그아웃 및 재로그인
가장 일반적인 해결 방법입니다. IDE 하단의 Accounts 아이콘을 클릭하여 로그아웃한 뒤, 다시 로그인합니다. 특히 라이선스가 개인에서 기업용으로 변경된 직후에 반드시 수행해야 합니다.

### 2단계: GitHub 테넌트 및 조직 정책 확인
기업용 계정의 경우, 관리자가 Copilot Access를 활성화했는지 확인해야 합니다. 
(참고: GitHub 프로필 설정 - Organizations - 해당 조직의 Copilot 설정 확인)

### 3단계: 네트워크 및 프록시 설정 점검
회사 망이나 VPN을 사용하는 경우, GitHub 서버와의 통신이 차단될 수 있습니다. 
* **필수 허용 도메인:** github.com, api.github.com, copilot-proxy.githubusercontent.com
* **SSL 인증서:** 회사에서 SSL 검사(Inspection)를 수행하는 경우, IDE에서 자체 인증서 설정을 조정해야 할 수 있습니다.

### 4단계: 확장 프로그램 버전 업데이트
이전 버전의 Copilot 확장 프로그램은 최신 인증 방식을 지원하지 않을 수 있습니다. 항상 최신 버전으로 유지하십시오.

### 5단계: 호스트 파일 및 방화벽 확인
특정 환경에서 hosts 파일 변조나 방화벽 규칙으로 인해 인증 토큰 교환이 실패할 수 있습니다. GitHub Copilot 출력 창(Output)의 로그를 통해 401(Unauthorized) 또는 403(Forbidden) 에러 발생 여부를 점검합니다.
