# GitHub Copilot을 활용한 레거시 코드 해독 및 분석

테스트 코드가 없고 복잡하게 얽힌 레거시 코드를 수정하기 전, GitHub Copilot을 강력한 분석 파트너로 활용하여 코드의 의도와 구조를 빠르게 파악하는 전략을 학습합니다.

---

## 1. 역설계 파트너 (Reverse Engineering Partner)

복잡하고 난해한 로직을 개발자가 이해할 수 있는 자연어로 번역하여 코드의 의도를 파악합니다.

* **코드 요약 및 의도 파악:** 수백 줄에 달하는 스파게티 코드를 선택하고 "이 로직을 비즈니스 관점에서 3줄로 요약해줘"라고 요청합니다. 이는 단순한 코드 해석을 넘어, 해당 코드가 해결하려는 실제 문제를 파악하는 데 도움을 줍니다.
* **데이터 흐름 추적:** 특정 변수가 어디서 생성되어 어떻게 변형되는지 질문하여 데이터의 생명 주기를 역으로 추적합니다.

---

## 2. 의존성 맵 생성 및 호출 관계 파악

레거시 코드의 가장 큰 위험 요소인 '강한 결합도'를 파악하기 위해 클래스와 함수 간의 호출 관계를 시각화하고 질문합니다.

* **영향도 분석:** "이 함수를 수정했을 때 영향을 받는 다른 모듈이나 함수 목록을 알려줘"와 같은 질문을 통해 수정 시 발생할 수 있는 부작용(Side Effect) 범위를 사전에 정의합니다.
* **외부 의존성 식별:** @workspace를 활용하여 "이 프로젝트에서 외부 API나 데이터베이스에 직접 의존하고 있는 지점들을 모두 찾아줘"라고 요청함으로써, Michael Feathers가 강조한 '접합부(Seams)'를 찾기 위한 기초 데이터를 확보합니다.



---

## 3. 코드 스멜(Code Smell) 식별 및 기술 부채 시각화

주관적인 판단이 아닌, 객관적인 기준에 근거하여 리팩토링이 시급한 지점을 찾아내고 기술 부채를 구체화합니다.

* **우선순위 선정:** "이 파일에서 클린 코드 원칙을 가장 크게 위반하고 있는 코드 스멜 3가지만 짚어주고, 리팩토링 우선순위를 제안해줘"라고 요청합니다.
* **복잡도 시각화:** 함수가 너무 길거나(Long Method), 클래스가 너무 많은 책임을 가지고 있는지(God Object)를 식별하여 리팩토링의 타겟을 명확히 합니다.

---

## 4. 실전 분석 프롬프트 사례

강의 현장에서 수강생들이 직접 활용할 수 있는 실전 프롬프트 예시입니다.

* **분석 질문 1:** "이 함수의 순환 복잡도(Cyclomatic Complexity)가 높은 이유를 설명하고, 논리적으로 분리 가능한 지점을 추천해줘."
* **분석 질문 2:** "이 클래스에서 객체 지향의 5대 원칙(SOLID) 중 위반되고 있는 항목을 찾아서 설명해줘."
* **분석 질문 3:** "이 레거시 코드에서 테스트 코드를 작성하기 위해 의존성을 분리(Decoupling)해야 하는 가장 핵심적인 위치는 어디야?"

---

## 5. 분석 세션 실습 가이드

1.  **대상 선정:** 프로젝트 내에서 가장 이해하기 어렵거나 오류가 자주 발생하는 레거시 파일을 엽니다.
2.  **의도 파악:** Copilot Chat에게 `/explain` 명령어를 사용하여 코드의 전체적인 비즈니스 로직을 파악합니다.
3.  **코드 스멜 식별:** 분석 프롬프트를 사용하여 리팩토링이 필요한 후보군을 리스트업합니다.
4.  **전략 수립:** 파악된 '접합부'를 중심으로 어떻게 테스트 코드를 먼저 입힐 것인지 계획을 세웁니다.

---

## 6. 핵심 포인트
* 코드 분석은 '수정'을 위한 전 단계이며, Copilot은 분석 시간을 획기적으로 단축해주는 도구임을 인지합니다.
* AI의 분석 결과를 맹신하기보다, 이를 바탕으로 실제 접합부를 찾아내고 테스트 코드를 설계하는 엔지니어링 능력을 기릅니다.
