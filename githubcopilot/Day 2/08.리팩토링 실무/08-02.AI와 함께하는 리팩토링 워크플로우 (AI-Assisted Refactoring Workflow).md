### AI와 함께하는 리팩토링 워크플로우 (AI-Assisted Refactoring Workflow)

AI는 단순히 코드를 고쳐주는 도구를 넘어, 리팩토링의 전 과정을 함께하는 '페어 프로그래밍' 파트너입니다. 안전하고 효율적인 리팩토링을 위한 4단계 표준 워크플로우를 학습합니다.

---

#### 1단계: 진단 (Diagnosis) - 코드 스멜 탐지
단순히 코드를 고치기 전, 무엇이 문제인지 AI의 객관적인 시각으로 분석합니다.
* **활동:** Copilot Chat에게 대상 메서드의 가독성이나 복잡도를 평가해달라고 요청합니다.
* **프롬프트 예시:** "이 메서드의 가독성을 저해하는 구체적인 요소가 뭐야? 클린 코드 원칙(Clean Code) 관점에서 짚어줘."

#### 2단계: 제안 (Suggestion) - 개선안 도출
진단 결과를 바탕으로 구체적인 수정 코드를 제안받습니다.
* **활동:** /refactor 명령어를 사용하거나 구체적인 기법(메서드 추출 등)을 명시하여 요청합니다.
* **프롬프트 예시:** "/refactor 이 메서드를 '메서드 추출' 기법을 사용해서 가독성 있게 쪼개줘. 중첩된 if 문은 Early Return 패턴으로 변경해줘."



#### 3단계: 검토 (Review) - 비즈니스 맥락 확인
AI는 코드 구조는 잘 파악하지만, 해당 산업군의 도메인 용어나 특수한 비즈니스 맥락은 놓칠 수 있습니다.
* **활동:** AI가 제안한 변수명이나 메서드 이름이 현재 프로젝트의 용어 사전(Glossary)과 일치하는지, 비즈니스 의도를 충분히 반영하는지 사람이 최종 판단합니다.
* **체크포인트:** "AI가 제안한 calc_amt라는 이름보다 우리 도메인에서 사용하는 calculate_invoice_total이 더 명확하지 않은가?"

#### 4단계: 검증 (Verification) - 회귀 테스트 (Regression Test)
리팩토링의 가장 중요한 단계입니다. 코드의 구조는 바뀌었어도 외부로 드러나는 기능은 동일해야 합니다.
* **활동:** 리팩토링 전 단계에서 작성해둔 테스트 코드를 실행합니다.
* **결과:** 모든 테스트가 Pass되는지 확인하여 의도치 않은 기능 변화(Regression)가 없음을 보장합니다.

---

#### 핵심 포인트
* **신뢰하되 검증하라 (Trust but Verify):** AI의 제안은 항상 출발점일 뿐이며, 최종 승인은 개발자의 몫입니다.
* **테스트 우선:** 테스트 코드가 없는 리팩토링은 리팩토링이 아니라 '도박'입니다. 반드시 검증 단계를 거쳐야 합니다.
* **작은 단계의 반복:** 한꺼번에 큰 규모를 리팩토링하기보다, AI와 함께 작은 단위로 쪼개어 수시로 검증하며 진행합니다.
