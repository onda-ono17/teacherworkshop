### 실습: GitHub Copilot을 활용한 단계별 리팩토링

본 실습은 앞서 제공된 OrderProcessor의 '나쁜 코드'를 GitHub Copilot과 함께 점진적으로 개선하며, 클린 코드의 핵심 원칙인 가독성 향상과 중복 제거를 실천하는 과정입니다.

---

#### 1단계: 메서드 추출 (Extract Method) 챌린지

**미션:**
`process_order` 내부의 '데이터 검증' 로직과 '가격 계산' 로직을 각각 별도의 전용 메서드로 분리하여 주 메서드의 복잡도를 낮추십시오.

* **수행 방법:**
    1. `process_order` 함수 내의 검증 로직 블록을 선택합니다.
    2. 인라인 채팅(Windows: `Ctrl+I` / Mac: `Cmd+I`)을 실행합니다.
    3. **프롬프트:** "이 로직을 _validate_order와 _calculate_total 메서드로 추출해줘."
* **학습 포인트:**
    - 거대했던 하나의 메서드가 분리된 메서드들의 호출부만 남게 되면서, 마치 읽기 쉬운 '목차'처럼 변하는 과정을 확인합니다.
    - 각 기능이 이름(Validate, Calculate)을 가짐으로써 주석 없이도 의도가 명확해짐을 체감합니다.

---

#### 2단계: 중복 제거 (DRY - Don't Repeat Yourself)

**미션:**
`process_order`와 `process_refund`에 각각 흩어져 있는 동일한 가격 계산 로직을 하나로 통합하여 유지보수 지점을 단일화하십시오.

* **수행 방법:**
    1. 두 메서드를 모두 포함하는 영역을 드래그하거나 파일 전체를 대상으로 Copilot Chat을 활성화합니다.
    2. **프롬프트:** "두 메서드에 중복된 가격 계산 로직이 있어. 이를 공통으로 사용할 수 있는 private 메서드로 만들고 각 메서드에서 이 공통 메서드를 호출하도록 코드를 수정해줘."
* **학습 포인트:**
    - 중복 코드를 제거함으로써 향후 계산 공식이 변경될 때 발생할 수 있는 '수정 누락' 버그를 원천 차단하는 방법을 배웁니다.
    - 코드의 전체 라인 수가 줄어들고 응집도가 높아지는 것을 관찰합니다.

---

#### 3단계: 가독성 개선 (Rename & Simplify)

**미션:**
모호한 명칭을 정리하고, 깊은 중첩 구조(Nested if)를 제거하여 코드의 인지 부하를 최소화하십시오.

* **수행 방법:**
    1. 리팩토링된 코드를 선택하고 다시 인라인 채팅을 실행합니다.
    2. **프롬프트:** "이 코드를 가드 클로즈(Guard Clauses) 패턴을 사용하여 가독성을 높여줘. 또한 total, t, data와 같은 모호한 변수명은 클린 코드 원칙에 따라 의미 있는 이름으로 수정해."
* **학습 포인트:**
    - **가드 클로즈 패턴:** 예외 상황을 메서드 상단에서 미리 쳐내어(Return early), 코드의 주 로직이 들여쓰기 없이 평탄하게 유지되는 효과를 확인합니다.
    - **명명 규칙:** AI가 제안하는 변수명이 실제 비즈니스 도메인(예: order_amount, line_item_total 등)을 얼마나 잘 반영하는지 평가합니다.
