# 실습: 장황하고 중복된 샘플 프로젝트 (OrderProcessor)

본 자료는 장황한 메서드(Long Method)와 중복 코드(Duplicate Code)라는 두 가지 핵심 코드 스멜을 포함하고 있는 실습용 소스 코드입니다. 수강생들은 이 코드를 분석하고 GitHub Copilot과 함께 리팩토링을 수행합니다.

---

#### 1. 코드 스멜 분석 포인트

제공된 OrderProcessor 클래스에서 다음의 문제점을 식별하십시오.

* 장황한 메서드 (Long Method): process_order 함수 하나에 검증, 계산, 할인, 입출력(I/O) 로직이 모두 포함되어 있어 가독성이 떨어지고 책임이 비대합니다.
* 중복 코드 (Duplicate Code): 주문 처리(process_order)와 환불 처리(process_refund) 메서드 내부에 동일한 가격 계산 로직이 중복되어 존재합니다.
* 응집도 부족: 비즈니스 규칙(할인율 등)과 저수준의 입출력 로직(프린트, DB 저장)이 섞여 있어 테스트와 재사용이 어렵습니다.

---

#### 2. 실습용 소스 코드
```
파일명: order_processor.py

class OrderProcessor:
    def process_order(self, order):
        # 1. 검증 (Validation)
        if order['items'] is None or len(order['items']) == 0:
            print("Error: No items")
            return
        if order['user_id'] is None:
            print("Error: No user")
            return
            
        # 2. 가격 계산 (Calculation) - 중복 발생 지점
        total = 0
        for item in order['items']:
            total += item['price'] * item['qty']
        
        # 3. 할인 로직 (비즈니스 규칙의 파편화)
        if order['type'] == 'VIP':
            total = total * 0.8
        elif order['type'] == 'NEW':
            total = total * 0.95
            
        # 4. 결제 및 알림 (I/O)
        print(f"User {order['user_id']} paid {total}")
        # ... (DB 저장 및 이메일 로직 생략) ...

    def process_refund(self, refund):
        # [코드 스멜] 가격 계산 로직이 process_order와 중복됨
        total = 0
        for item in refund['items']:
            total += item['price'] * item['qty']
        
        # ... 환불 전용 로직 수행 ...
        print(f"Refund of {total} processed")
```

#### 3. 리팩토링 실습 미션

수강생은 GitHub Copilot과 협업하여 다음의 단계를 수행합니다.

단계 1: 메서드 추출 (Extract Method)
* 활동: process_order 내의 가격 계산 로직을 별도의 함수(예: calculate_total)로 추출합니다.
* 프롬프트: "process_order와 process_refund에 중복된 가격 계산 로직을 calculate_total이라는 별도의 메서드로 추출해줘."

단계 2: 검증 로직 분리
* 활동: 입력 데이터의 유효성을 검사하는 부분을 별도의 검증 전용 메서드로 분리합니다.
* 프롬프트: "주문 데이터의 유효성을 검사하는 부분을 validate_order 메서드로 분리하고, 실패 시 적절한 예외(Exception)를 던지도록 수정해줘."

단계 3: 비즈니스 규칙 상수화 및 클린 코드 적용
* 활동: 하드코딩된 할인율을 상수나 정책 클래스로 분리하여 가독성을 높입니다.
* 프롬프트: "할인율 수치들을 상수로 선언하고, if-elif 문을 더 깔끔한 방식(예: Dictionary Mapping)으로 리팩토링해줘."

---

#### 4. 핵심 포인트

* 재사용성의 발견: 중복된 로직을 하나의 메서드로 합치는 것만으로도 버그 수정 포인트가 절반으로 줄어든다는 점을 확인합니다.
* 주석 없는 코드: 메서드 추출을 통해 각 로직에 이름(validate, calculate 등)이 붙으면서 코드의 가독성이 향상됨을 관찰합니다.
* 회귀 테스트: 리팩토링 전후로 동일한 입력값에 대해 동일한 결과값이 나오는지 테스트 코드를 통해 반드시 검증해야 함을 강조합니다.
