### 코드 스멜(Code Smell)과 처방전

리팩토링의 대가들이 정의한 '코드 스멜'은 프로그램의 작동에는 문제가 없으나, 유지보수를 어렵게 만들고 잠재적인 버그를 품고 있는 코드의 징후를 의미합니다. Refactoring.Guru의 가이드라인을 바탕으로 가장 빈번하게 발생하는 두 가지 병목 현상과 그 해결책을 학습합니다.

---

#### 1. 장황한 메서드 (Long Method)

* **증상:** 하나의 메서드가 수십, 수백 줄에 달하며 너무 많은 일을 수행합니다. 코드를 읽다가 앞부분의 맥락을 잊어버리게 되며, 특정 로직만 재사용하기가 불가능합니다.
* **처방 - 메서드 추출(Extract Method):** - 메서드 내부의 응집된 로직을 별도의 작은 메서드로 분리합니다.
    - **효과:** 분리된 메서드에 명확한 이름을 붙임으로써, 코드 자체가 주석 역할을 하게 되어 가독성이 비약적으로 향상됩니다.
* **Copilot 활용:** - "이 함수에서 독립적인 기능을 수행하는 부분을 별도의 메서드로 추출해줘. 각 메서드의 이름은 기능을 명확히 나타내야 해."



---

#### 2. 중복 코드 (Duplicate Code)

* **증상:** 유사하거나 동일한 로직이 프로젝트 곳곳에 흩어져 있습니다. 비즈니스 규칙이 변경될 때 모든 곳을 찾아 수정해야 하며, 하나라도 누락할 경우 시스템 전체의 데이터 불일치나 버그가 발생합니다.
* **처방 - 함수 공통화 및 분리:**
    - **메서드 올리기(Pull Up Method):** 서브 클래스들에 중복된 메서드가 있다면 슈퍼 클래스로 이동시킵니다.
    - **유틸리티화:** 클래스에 종속되지 않는 공통 로직은 유틸리티(Utility) 클래스나 헬퍼(Helper) 함수로 분리하여 관리합니다.
* **Copilot 활용:** - "@workspace 이 프로젝트 내에서 이 로직과 중복되는 코드가 있는지 찾아줘. 발견된다면 이를 공통 함수로 어떻게 통합할 수 있을지 제안해줘."



---

#### 3. 리팩토링 프롬프트 전략

실제 코드 스멜을 제거할 때 사용할 수 있는 구체적인 지시어입니다.

* **가독성 중심:** "이 함수의 인지 복잡도(Cognitive Complexity)를 낮추기 위해 로직을 쪼개줘. 제어문(if/for)이 너무 깊게 중첩되어 있어."
* **유지보수 중심:** "이 클래스에 구현된 날짜 계산 로직을 다른 곳에서도 쓸 수 있게 유틸리티 함수로 분리해줘."
* **단일 책임 원칙(SRP) 중심:** "이 함수는 데이터 계산과 DB 저장을 동시에 하고 있어. 계산 로직만 담당하는 순수 함수(Pure Function)로 분리해줘."

---

#### 4. 핵심 포인트

* **냄새를 맡는 능력:** 코드 스멜을 식별하는 것은 기술적인 도구보다 '개발자의 감각'이 중요합니다. 나쁜 코드의 징후를 느끼는 즉시 리팩토링을 고려해야 합니다.
* **점진적 개선:** 한꺼번에 모든 스멜을 제거하려 하지 말고, 메서드 추출과 같은 작은 단위부터 시작하여 테스트 코드가 통과하는 것을 확인하며 전진합니다.
* **주석의 대체:** "주석이 필요한 곳이 바로 메서드로 추출해야 할 지점"이라는 격언을 상기하며, 이름만으로 의도가 전달되는 코드를 지향합니다.
